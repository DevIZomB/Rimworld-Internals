RimWorld 1.6.4633 — Shared Simulation/Interface State

Static analysis + AI-assisted code review of 8,687 decompiled C# files for multiplayer sync

Why This Matters for Multiplayer

In multiplayer, the game simulation must be deterministic — identical inputs produce identical state on every client. Desyncs happen when:

  1. UI code writes simulation state — each client's UI runs independently, so local-only writes corrupt shared state
  2. Rand state diverges — a single unmatched Rand.Value call on one client shifts ALL subsequent random numbers
  3. Floating-point accumulation drifts — workLeft -= speed × delta accumulates tiny CPU-specific errors over thousands of ticks
  4. Camera-dependent tick rates — Pawn.TickInterval varies by camera distance, meaning different camera positions = different tick frequencies

Top Desync Risks — Ranked

CRITICAL Camera-Based Tick Intervals

Pawn.TickInterval and Thing.DoTick use UpdateRateTicks which depends on camera distance. Different camera positions across clients → different tick frequencies → tracker state diverges.

Fields: All Pawn tracker states (needs, health, jobs, mind)
Fix: Patch TickInterval to use fixed delta in MP
AI — High Confidence

CRITICAL Global Rand State

Rand.seed + Rand.iterations are static globals. A single extra Rand call on one client desyncs every future random number for the entire game session.

Biggest consumers: Storyteller, Interactions (6+ calls/event), Fire (8 calls/150t/fire), Construction
Fix: All Rand-consuming decisions must sync via Commands
AI — High Confidence

CRITICAL UI Toggles That Modify SIM State

Gizmo buttons directly write fields that Tick logic reads: holdFire, holdOpenInt, autoRearm, autoAgeReversal, etc. Without MP command sync, each client sets these independently.

Found: 44 critical + 12 high risk classes in static scan
Fix: All Gizmo state changes must route through MP command layer

HIGH Unseeded Weighted Random

Combat target selection (GetRandomShootingTargetByScore), job selection from ThinkTree, and Toils_Recipe.DoRecipeWork ingredient selection all use TryRandomElementByWeight without visible seed isolation.

Impact: Different target/job picks → divergent game state

HIGH Floating-Point Accumulation

workLeft -= speed × delta in crafting, Temperature += equalization in rooms, severity += change in hediffs — all accumulate tiny CPU-specific rounding errors over thousands of ticks.

Fix: Periodically resync float values between clients, or convert to fixed-point
AI — High Confidence

HIGH HashCode Non-Determinism

TickList distributes Things into tick-buckets by GetHashCode(). .NET does not guarantee identical HashCodes across processes — different bucket assignments → different tick order.

Fix: Use deterministic hashing (thingIDNumber, position-based)
AI — High Confidence

MEDIUM Lazy Init With Rand

Zone.Cells shuffles on first property access. If clients access at different times → Rand consumed at different points in the sequence → divergence.

Fix: Eagerly initialize at known sync points
AI — High Confidence

MEDIUM Shared Material Mutation

Graphic_Multi_AgeSecs calls Material.SetFloat() on shared materials during Draw phase. Not thread-safe and can cause visual inconsistency across clients.

AI — High Confidence

Culture/Locale Issue AI — High Confidence

LongEventHandler async operations must call CultureInfoUtility.EnsureEnglish() on worker threads. Without this, non-English systems parse floats differently (comma vs decimal point). MP async operations that parse numbers on background threads will produce different values on different locale systems.
