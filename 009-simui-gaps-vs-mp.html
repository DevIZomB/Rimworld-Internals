CRITICAL IncidentWorker_SeasonalFlooding.CanFireNowSub

Type: SIM↔UI — per-client UI state in SIM method

File: RimWorld/IncidentWorker_SeasonalFlooding.cs (RI-71)

Vector2 longLat = Find.WorldGrid.LongLatOf(Find.WorldSelector.SelectedTile);
Season season = GenDate.Season(Find.TickManager.TicksGame, longLat);

Problem: CanFireNowSub() is called by the Storyteller in SIM context. Find.WorldSelector.SelectedTile is per-client UI state — each player can have a different tile selected on the world map. Different tiles → different hemisphere → different season → incident fires on one client but not the other.

MP Mod Status: No patch found. SelectedTile is only referenced in the MP mod in FactionSidebar.cs for faction setup, not for this incident.

Note: Likely a vanilla bug. Should use map.Tile instead of Find.WorldSelector.SelectedTile. Only relevant when RW 1.6 Seasonal Flooding is active.

Suggested fix: Transpiler replacing Find.WorldSelector.SelectedTile with parms.target.Tile, or a prefix that temporarily sets SelectedTile to the map tile.

HIGH PitGate.Tick — Rand behind CurrentMap check

Type: RNG ROOT — Rand consumption depends on camera

File: RimWorld/PitGate.cs (RI-89), line 139

// Inside if (IsCollapsing) / CollapseStage == 2:
if (Find.CurrentMap == base.Map && Rand.MTBEventOccurs(2f, 60f, 1f))
{
    Find.CameraDriver.shaker.DoShake(0.2f);
}

Problem: Short-circuit evaluation: Rand.MTBEventOccurs is ONLY called when the client is viewing this map. Client A sees the map → consumes Rand. Client B sees a different map → skips Rand. Global Rand counter diverges → all future random numbers are shifted.

MP Mod Status: No patch for this specific Rand call. UndercaveMapComponent.MapComponentTick is patched (Determinism.cs:541), but PitGate.Tick itself has no determinism patch. Only debug gizmos are synced (SyncMethods.cs:462-464).

Note: Only affects Anomaly DLC, only during PitGate collapse stage 2. Impact depends on MTB event frequency.

Suggested fix: Always call Rand.MTBEventOccurs, only conditionally DoShake: bool shake = Rand.MTBEventOccurs(2f, 60f, 1f); if (Find.CurrentMap == base.Map && shake) DoShake(...);

HIGH WorldObject.Print — Rand without PushState

Type: RNG ROOT — Rand in render method without isolation

File: RimWorld.Planet/WorldObject.cs (RI-45), line 514-519

public virtual void Print(LayerSubMesh subMesh)
{
    float num = Rand.RangeSeeded(0f, 0.01f, ID);  // ← SAFE (seeded)
    WorldRendererUtility.PrintQuadTangentialToPlanet(
        DrawPos, 0.7f * averageTileSize, 0.03f + num, subMesh,
        counterClockwise: false, Rand.Range(0f, 360f));  // ← NOT seeded!
}

Problem: Print() builds world meshes and is called during the render pass when a layer is dirty. The mesh rebuild happens per-client at different times. Rand.Range(0f, 360f) consumes the global Rand counter without PushState.

MP Mod Status: No patch for WorldObject.Print or the Rand call within it. The VTR patch (VTRSyncPatch.cs) synchronizes the tick rate, but not the Rand consumption in the render pipeline.

Note: Rand.RangeSeeded(0f, 0.01f, ID) on the same line is correctly isolated. Only the rotation Rand is the problem. Affects all WorldObjects that don't override Print().

Suggested fix: Replace Rand.Range(0f, 360f) with Rand.RangeSeeded(0f, 360f, ID * 7) (analogous to the existing seeded call above), or wrap the entire Print() in PushState/PopState.
