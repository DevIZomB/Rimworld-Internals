How to Test These Fixes

Zetrith's MP mod has a built-in desync detection system that compares Rand.iterations and game state hashes between clients every tick. Each gap below includes a specific "How to Trigger" scenario. The general testing workflow:

  1. Run the trigger scenario without the fix — if the gap is real, the desync report shows Rand.iterations divergence.
  2. Apply the fix, run the same scenario — if the desync disappears, the fix works.
  3. Verify no side effects — gameplay should be unchanged (trees still drop leaves, buildings still fail, etc.).

Quick single-player validation: Log Rand.iterations before and after the patched method. If unchanged after PushState/PopState, the isolation works — zero global Rand consumed.

P1 — Plant.TickLong: Leaf Motes Consume Global Rand PRACTICAL HIGH

Plants with dropLeaves=true call Rand.InsideUnitCircleVec3, Rand.Value, and Rand.Range directly in TickLong() for leaf particles. Not through MoteMaker, not wrapped in PushState. ~3 Rand calls per tree per TickLong (every 2000 ticks). A colony with 50 trees = 150 unprotected Rand calls.

Triggers when: Any colony with trees (very common). If one client has a tree destroyed by fire that the other doesn't, Rand diverges permanently.

RimWorld/Plant.cs (TickLong, dropLeaves section)
Suggested Fix
[HarmonyPatch(typeof(Plant), nameof(Plant.TickLong))]
static class PlantLeafMotesPatch
{
    static void Prefix(Plant __instance)
    {
        if (Multiplayer.Client != null && __instance.def.plant.dropLeaves)
            Rand.PushState(__instance.thingIDNumber);
    }

    static void Finalizer(Plant __instance)
    {
        if (Multiplayer.Client != null && __instance.def.plant.dropLeaves)
            Rand.PopState();
    }
}

Isolates all Rand calls inside TickLong per plant. The thingIDNumber seed ensures deterministic-per-plant results without affecting the global Rand sequence.

Why this should work: Rand.PushState(seed) saves the current global Rand state to a stack, then sets a new seed. All Rand calls between Push and Pop use the new seed. PopState() restores the exact previous state — as if the calls in between never happened. The global iterations counter is unchanged.

How I arrived at this fix: Zetrith uses the exact same pattern in Seeds.cs:192-210 for PawnRenderer.SetAllGraphicsDirty — another case where visual code consumes global Rand during ticks. They wrap it in PushState(Gen.HashCombineInt(pawn.thingIDNumber, ConstantRandSeed)). The plant leaf mote problem is structurally identical: visual Rand in tick code, no gameplay effect. Evidence: Zetrith Seeds.cs line 200 — SeedPawnGraphics uses PushState(thingIDNumber) for the same class of problem.
How to Trigger This Desync
  1. Start an MP game on a map with a temperate forest biome (many trees with dropLeaves).
  2. Use dev mode: spawn ~50 extra oak/poplar trees in a cluster.
  3. On one client only, use dev mode to destroy 5–10 trees (Destroy tool, not the other client).
  4. Let the game run for 2000+ ticks (one TickLong cycle).
Expected result: Desync report shows Rand.iterations divergence after the next TickLong cycle, because the destroyed trees no longer consume their 3 Rand calls on one client but still do on the other.

P3 — SnowGrid: Mesh Dirty Throttle Uses Global Rand PRACTICAL HIGH

Rand.Value < 0.0125f throttles mesh updates on small snow depth changes (<0.15). During snowfall, hundreds of cells trigger this per tick. Not in PushState.

Triggers when: Snowfall on any map (extremely common in temperate/cold biomes). Hundreds of Rand calls per tick during snow events.

Verse/SnowGrid.cs
Suggested Fix
[HarmonyPatch(typeof(SnowGrid),
    "CheckVisualOrPathCostChange")]
static class SnowGridRandPatch
{
    // Replace Rand.Value with deterministic hash
    static IEnumerable<CodeInstruction> Transpiler(
        IEnumerable<CodeInstruction> insts)
    {
        var randValue = AccessTools.PropertyGetter(
            typeof(Rand), nameof(Rand.Value));

        foreach (var ci in insts)
        {
            if (ci.Calls(randValue))
            {
                // Replace: Rand.Value < 0.0125
                // With: (cellIndex.GetHashCode() % 80 == 0)
                yield return new CodeInstruction(
                    OpCodes.Ldarg_1); // cell index
                yield return new CodeInstruction(OpCodes.Call,
                    AccessTools.Method(typeof(SnowGridRandPatch),
                        nameof(DeterministicThrottle)));
                continue;
            }
            yield return ci;
        }
    }

    static float DeterministicThrottle(int cellIndex)
    {
        // Returns 0.0 (below threshold) ~1/80 of the time
        return (cellIndex.GetHashCode() % 80 == 0) ? 0f : 1f;
    }
}

Replaces probabilistic throttle with a deterministic hash-based one. Same ~1.25% trigger rate, zero Rand consumption.

Why this should work: The original code uses Rand.Value < 0.0125 purely as a performance throttle — it decides whether to update a mesh, not any gameplay state. Replacing it with cellIndex % 80 == 0 gives the same ~1.25% trigger rate but uses zero Rand calls. The mesh update is visual-only; which specific cells get updated first doesn't affect simulation.

How I arrived at this fix: This is a different pattern from PushState. PushState would work too, but it's wasteful — hundreds of cells trigger this per tick during snowfall, each needing a Push+Pop. A hash replacement is cheaper and cleaner. Ludeon themselves use hash-based patterns elsewhere (e.g., IsHashIntervalTick for throttling tick checks). This follows that same principle: deterministic throttle via hash instead of probabilistic throttle via Rand. Evidence: Verse/GenTicks.cs — IsHashIntervalTick uses thingIDNumber hash for deterministic throttling. Same concept applied to cell-based throttling.
How to Trigger This Desync
  1. Start an MP game on a map with cold/temperate biome that gets snowfall.
  2. Wait for a snowfall event (or use dev mode: Set weather > SnowHard).
  3. Speed up to 3x and let snow accumulate for 1–2 in-game hours.
  4. The hundreds of small snow-depth changes per tick each call Rand.Value for mesh throttling.
Expected result: If any prior state difference exists (even tiny), the Rand.Value calls during snowfall amplify the divergence rapidly. Monitor Rand.iterations — during heavy snowfall they should increase by 100+ per tick from this source alone. Any prior 1-iteration offset becomes a cascade.

P12 — InspirationHandler: Mood-Gated MTB + RandomElementByWeight PRACTICAL MEDIUM

Every 100 ticks per pawn: mood gate (>0.5), then Rand.MTBEventOccurs (1–2 calls), on success RandomElementByWeightWithFallback (N+1 calls, N = available inspirations). The mood gate is a hard cutoff — slightly different mood = completely different Rand behavior.

Triggers when: Colonists with mood above 50% (very common). Slight mood calculation differences = one client makes the Rand call, the other skips it entirely.

[HarmonyPatch(typeof(InspirationHandler),
    "CheckStartRandomInspiration")]
static class InspirationRandPatch
{
    static void Prefix(InspirationHandler __instance)
    {
        if (Multiplayer.Client != null)
            Rand.PushState(Gen.HashCombineInt(
                __instance.pawn.thingIDNumber,
                Find.TickManager.TicksGame / 100));
    }

    static void Finalizer()
    {
        if (Multiplayer.Client != null)
            Rand.PopState();
    }
}

Seeds with pawn ID + tick bucket (100-tick intervals). Even if mood-gate differs, the global Rand is unaffected.

Why this should work: The mood gate (>0.5) is a hard cutoff. If mood differs by 0.001 between clients, one client enters the MTBEventOccurs + RandomElementByWeight block and the other doesn't. With PushState, both paths consume zero from the global Rand. The pawn ID ensures deterministic results per pawn, the tick bucket (TicksGame / 100) ensures different results each check interval.

How I arrived at this fix: Zetrith uses this exact pattern for PreceptComp_UnwillingToDo_Chance.MemberWillingToDo (Seeds.cs:212-236) — another case where a UI/mood-dependent check can call Rand from different code paths. Their fix: PushState(Gen.HashCombineInt(pawnId, TicksGame)). The inspiration handler has the same structure: mood-gated Rand calls that may or may not execute. Evidence: Zetrith Seeds.cs line 212 — SeedPreceptComp_UnwillingToDo_Chance uses PushState(pawnID ^ tick) for a mood/belief-gated Rand call. InspirationHandler is the same pattern.
How to Trigger This Desync
  1. Start an MP game with 5+ colonists.
  2. On one client only, use dev mode to set one colonist's mood to exactly 0.51 (just above the 0.5 gate). On the other client, set the same colonist's mood to 0.49 (just below).
  3. Wait for the 100-tick inspiration check cycle.
Expected result: Client with mood 0.51 enters the MTBEventOccurs block (1–2 Rand calls), client with mood 0.49 skips it entirely (0 Rand calls). After one check cycle, Rand.iterations diverges by 1–2. If the MTB succeeds, RandomElementByWeight adds N+1 more divergent calls.

Why Document Theoretical Gaps?

Theoretical gaps become practical when a prior desync has already occurred. Once game state diverges even slightly (e.g., from a practical gap above), the theoretical gaps become amplifiers that accelerate the divergence. Fixing the practical gaps first prevents the cascade that makes theoretical gaps trigger.

Think of it as: Practical gaps = root causes. Theoretical gaps = accelerators. Fix the root causes first, and most theoretical gaps never fire.
