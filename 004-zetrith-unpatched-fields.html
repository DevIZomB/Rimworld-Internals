1. Visual Rand Consuming Global State in Tick Code UNPATCHED HIGH RISK

Zetrith patches MoteMaker.MakeStaticMote for deterministic mote spawning. But several systems call Rand directly in tick code for visual effects without going through MoteMaker. These consume global Rand state and can cause desync if any condition makes one client skip the call.

Plant.TickLong — Leaf Drop Motes HIGH

Plants with dropLeaves=true call Rand.InsideUnitCircleVec3, Rand.Value, and Rand.Range for leaf particle positions/scales in TickLong(). These are not wrapped in PushState and not routed through MoteMaker. With many trees, this is a significant Rand consumer (~3 calls per tree per TickLong).

Impact: If a plant exists on one client but not the other (e.g., destroyed by fire on one side), the Rand sequence diverges permanently.

Fix: Wrap leaf mote code in Rand.PushState(plant.thingIDNumber) / PopState().

RimWorld/Plant.cs (TickLong, dropLeaves section)
MentalState_BabyCry — Tear Visuals in MentalStateTick HIGH

Baby crying uses 4–6 Rand calls per tear cycle (25–40 ticks): randAngle.RandomInRange, randScale.RandomInRange, TicksBetweenTears.RandomInRange, Rand.Value × 180 for mote rotation. All in MentalStateTick() — gameplay tick, not draw code.

Impact: Multiple babies = dozens of unprotected Rand calls per tick cycle. If one client has a different baby count or mental state, permanent Rand drift.

Fix: PushState(pawn.thingIDNumber) around tear-visual code in MentalStateTick.

Verse.AI/MentalState_BabyCry.cs:34-56
SnowGrid — Mesh Dirty Throttle MEDIUM

CheckVisualOrPathCostChange uses Rand.Value < 0.0125f to throttle mesh updates on small depth changes (<0.15). This is a rendering optimization that consumes global Rand. During snowfall, hundreds of cells trigger this per tick.

Impact: Variable Rand calls based on snow depth changes. If snow state diverges slightly → different call count → cascading desync.

Fix: Replace Rand.Value with Rand.ValueAsync(cellIndex) or use cell hash modulo.

Verse/SnowGrid.cs
Fire.SpawnSmokeParticles — Smoke Position Rand MEDIUM

Fire tick calls SpawnSmokeParticles which uses 1–2 Rand calls for smoke positions. Combined with DoComplexCalcs (5+ calls/150 ticks) and TrySpread (4 calls), a single fire consumes ~8 Rand calls per 150 ticks. 50 fires = 400+ unprotected Rand calls.

Impact: If fire spreads differently on one client (due to prior state divergence), the Rand consumption differs enormously.

Fix: PushState(fire.thingIDNumber) around smoke particle code. TrySpread and DoComplexCalcs must remain global (they affect simulation).

Verse/Fire.cs

2. Constructor Rand Before Lord PushState UNPATCHED HIGH RISK

Zetrith's async system seeds Lord-creation via Lord.SetJob with PushState(loadID * 193). But some LordJob constructors consume Rand before the Lord is created — meaning those calls hit the global unprotected Rand state.

LordJob_Joinable_Party — Duration in Constructor HIGH

Rand.RangeInclusive(5000, 15000) is called in the constructor to set party duration. This happens before Lord.SetJob's PushState. If two clients create the party at different ticks, the Rand call happens at different positions in the global sequence.

Impact: Desync if party creation timing differs. The constructor also sets spot and organizer.

Fix: Harmony prefix on constructor to PushState with a deterministic seed (e.g., initiator pawn ID).

Verse.AI.Group/LordJob_Joinable_Party.cs (constructor)
LordToil_Breaching — preferMelee in UpdateAllDuties HIGH

Rand.Chance(0.5f) in UpdateAllDuties when breachDest is initialized. This runs in the normal tick flow (LordToilTick → UpdateAllDuties). If breachDest becomes invalid at different ticks on different clients → the Rand call happens at different simulation positions.

Impact: Different melee preference → different duty assignments → different pawn behavior → cascading desync.

Fix: Seed with Lord.loadID before the Rand.Chance call.

Verse.AI.Group/LordToil_Breaching.cs
LordToil_Sappers — EscortRadius per Pawn HIGH

EscortRadiusRanged/Melee.RandomInRange is called in UpdateAllDuties for every escort pawn. This happens at every duty reassignment. If a pawn dies on one client triggering Notify_ReachedDutyLocation → duty reset → different number of Rand calls.

Impact: Variable Rand count per client based on pawn survival. Escort radius affects pathfinding targets.

Fix: Use Rand.RangeSeeded(min, max, pawn.thingIDNumber ^ lord.loadID).

Verse.AI.Group/LordToil_Sappers.cs

3. Float-Point Accumulation Drift UNPATCHED MEDIUM RISK

RimWorld uses float extensively for temperature, work progress, and stat calculations. Zetrith doesn't address potential IEEE 754 rounding differences between clients. While typically negligible on identical hardware, these can accumulate over long play sessions.

Room Temperature Equalization MEDIUM

RoomTempTracker accumulates temperature from 4 equalization sources every tick. Each is a float multiplication + addition. Over thousands of ticks, tiny precision differences compound. Temperature affects: food rot rate, hypothermia/heatstroke, plant growth, animal behavior.

Impact: Slow drift over time. Temperature difference → different rot timing → different food availability → different pawn decisions → eventual desync.

Fix: Periodic temperature snap-sync (quantize to nearest 0.01 every N ticks) or server-authoritative temperature.

Verse/RoomTempTracker.cs
Recipe Work Progress MEDIUM

Toils_Recipe.DoRecipeWork: workLeft -= workSpeed × delta uses float subtraction every tick. Different FPU behavior could cause one client to finish a recipe 1 tick earlier, triggering CompleteRecipeWork (which consumes Rand for quality generation) at a different time.

Impact: Quality generation Rand call at different tick = permanent Rand state divergence.

Fix: Use fixed-point integer arithmetic for work progress, or snap workLeft periodically.

Verse/Toils_Recipe.cs
Stat Capacity Factor Lerp LOW

StatWorker uses Mathf.Lerp(num, num * factor, weight) for capacity factors. The result depends on evaluation order and float precision. Different Mono/.NET runtimes could produce slightly different results.

Impact: Unlikely with identical game versions, but possible with mixed Linux/Windows clients.

RimWorld/StatWorker.cs:384-391

Why These Matter Despite Lockstep
Zetrith's lockstep ensures all clients run the same ticks in the same order. These Rand calls are safe as long as game state is perfectly identical. The risk comes from any prior divergence (float drift, missed sync, mod difference) that causes one client to take a different code path — changing the number of Rand calls and permanently offsetting the global RNG state.

6. Incident Worker Rand Pipelines PARTIALLY PATCHED HIGH RISK

Zetrith syncs Storyteller settings and patches the tick system. But the incident execution pipelines themselves are massive Rand consumers that depend on perfectly synchronized state. If state diverges even slightly before an incident fires, the Rand consumption differs catastrophically.

PawnGenerator — The Nuclear Option NUCLEAR

A single PawnGenerator.GeneratePawn() call can consume 150–48,000 Rand calls due to 120-retry loops, 80-retry apparel generation, 500-iteration trait fitting, and recursive relation spawning. Zetrith does NOT wrap PawnGenerator in PushState.

Why it matters: Raids, traders, visitors, quests — all trigger PawnGenerator. A single retry-count difference between clients = thousands of Rand calls offset = guaranteed permanent desync.

Zetrith's implicit defense: Lockstep ensures the same incidents fire at the same tick. But if any prior state divergence causes different PawnGenerationRequest constraints (different faction, different world pawn count for redress chance), the retry count differs.
