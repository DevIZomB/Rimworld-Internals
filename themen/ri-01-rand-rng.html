Rand & RNG

Warnung
Jeder Rand-Aufruf in einem Mod verschiebt den globalen iterations-Counter. Für nicht-gameplay Zufall IMMER PushState/PopState verwenden!
#1 Globaler State Verse/Rand.cs:13-17

MP-Kritisch — Zwei Felder definieren den gesamten RNG-State: seed (uint) und iterations (uint). Jeder Value-Aufruf erhöht iterations um 1. Determinismus hängt davon ab, dass alle Clients exakt gleich viele Rand-Calls machen.

#2 MurmurHash Verse/MurmurHash.cs:1-34

Pure Hash-Funktion: GetInt(seed, iterations) → deterministischer int. Keine Seiteneffekte, kein State. Wird von Rand.Value intern verwendet.

#3 State Stack PushState/PopState Verse/Rand.cs:440-454

Stack-basiertes State-Management: PushState() speichert aktuellen seed+iterations auf Stack und setzt neuen Seed. PopState() stellt alten State wieder her. MUSS für UI/nicht-gameplay Rand verwendet werden.

#4 Value & Basis-Operationen Verse/Rand.cs:35-51

Rand.Value → float [0,1). Rand.Int, Rand.Bool, Rand.Range(min,max). Jeder Aufruf erhöht iterations. Rand.Chance(f) = Value < f.

#5 MTBEventOccurs Verse/Rand.cs:509-554

MP-Kritisch — Mean Time Between Events: Berechnet Wahrscheinlichkeit pro Tick basierend auf MTB-Tagen. Verbraucht einen Rand.Value-Call. Wird massiv genutzt (Krankheiten, Mental Breaks, Events).

#6 Gaussian, ByCurve, InsideUnitCircle Verse/Rand.cs:137-507

Erweiterte Zufallsfunktionen: Gaussian (Box-Muller, 2 Rand-Calls), ByCurve (gewichtete Kurve), InsideUnitCircle/InsideUnitCircleVec3 (Rejection Sampling, variable Rand-Calls).

InsideUnitCircle verbraucht variable Anzahl Rand-Calls je nach Rejection — besonders desync-gefährlich.
